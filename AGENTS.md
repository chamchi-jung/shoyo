# AGENTS.md

## 프로젝트 원칙

이 프로젝트는 커뮤니티 서비스보다 먼저 개인 취향을 전시하는 페이지를 만드는 것이 핵심이다. 초기 구현에서는 로그인, DB, 팔로우, 댓글, 좋아요를 추가하지 않는다.

## 기술 방향

- Next.js App Router와 TypeScript를 사용한다.
- 스타일링은 Tailwind CSS를 기본으로 사용한다.
- 프로필 데이터는 당분간 `src/data`의 하드코딩된 샘플 데이터로 관리한다.
- 프로필 화면은 `src/components/profile` 아래의 작은 컴포넌트로 나눈다.
- 새 화면을 만들 때는 데이터 타입을 먼저 확장하고, 그 타입이 UI에서 실제로 드러나게 한다.

## 디자인 방향

- 현대적인 SaaS 대시보드보다 개인 홈페이지, 취향 아카이브, 오래된 블로그 스킨의 느낌을 우선한다.
- 기능 설명용 랜딩 페이지보다 실제 프로필 화면을 먼저 보여준다.
- 카드와 섹션은 과하게 둥글게 만들지 않는다.
- 사용자의 취향 목록, 자기소개, 자유 텍스트, 방 안의 오브젝트가 화면의 주인공이 되게 한다.
- 샘플 프로필은 서로 다른 사람의 방처럼 보여야 한다.

## 커스터마이징 방향

- 사용자별 테마 값은 CSS 변수로 연결한다.
- 레이아웃은 우선 `zine`, `shelf`, `room` 같은 프리셋으로 검증한다.
- 추후 커스텀 CSS를 붙일 수 있도록 `ProfileTheme.customCss` 필드를 유지한다.
- 커스텀 CSS 적용은 보안과 격리 전략을 정한 뒤에 구현한다.

## MVP 경계

- 지금은 하드코딩된 샘플 데이터로 보이는 경험을 검증한다.
- 소셜 기능보다 프로필 전시 밀도, 편집 감각, 테마 차이를 먼저 만든다.
- `/studio/profile`은 저장 없는 편집 감각을 검증하는 공간이다.
- 서버 저장 전에는 `localStorage` 초안 저장까지만 사용한다.
- 좋아요/댓글/팔로우보다 방명록, 웹링, 랜덤 프로필 구경 같은 가벼운 방문 흔적을 나중에 검토한다.

## 개발 태도

- MVP 범위를 지키고, 요청되지 않은 소셜 기능을 먼저 만들지 않는다.
- 새 기능을 추가할 때는 데이터 타입, 라우트, 화면 컴포넌트의 책임을 분리한다.
- 변경 후에는 가능한 한 `npm run lint`와 `npm run build`로 확인한다.
